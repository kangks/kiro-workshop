## **09:00 \- 10:30 | Part 1: "Reverse Spec-ing" DVAIA via Kiro**

**Objective:** Use Kiro to map out the undocumented, vulnerable DVAIA codebase into structured requirements and design specs.

### **Detailed Steps:**

1. **Context Pinning:** Instruct attendees to use Kiro's UI to "pin" app.py, the /routes folder, and the LangChain agent tool configurations.  
2. **The Reverse-Spec Prompt:** Attendees prompt Kiro to read the code and document the current state.**Prompt (in Kiro):** "Analyze the pinned DVAIA codebase. We need to document its current state using a Design-first workflow. Generate a high-level design.md covering the Flask UI, LangChain orchestration, Ollama LLM integration, and Qdrant vector database. Do not change any code."  
3. **Deriving Tasks:** Once Kiro generates design.md, click **Move to Implementation Plan**. Kiro will generate tasks.md, detailing how the current architecture is wired.  
4. **Key Lesson:** In Kiro's Spec-Driven Development (SDD), you always establish the baseline design.md and tasks.md before writing a single line of code.

## **10:45 \- 12:30 | Part 2: The GenAI Security Gauntlet (AWS Security Agent)**

**Objective:** Use **AWS Security Agent** to execute an on-demand penetration test, then use Kiro's Bugfix Spec to remediate the findings agentically.

### **Detailed Steps:**

1. **The Autonomous Pen Test:** Attendees open the **AWS Security Agent Web App**.  
   * Navigate to "Penetration Tests" \-\> "Create Penetration Test".  
   * Provide the local/ngrok URL of the running DVAIA application and attach the design.md generated by Kiro in Part 1 to give the agent context.  
   * Click **Execute**.  
2. **Analyze Findings:** The AWS Security Agent autonomously explores the app and reports a critical finding: The Agentic Testing panel uses an intentionally vulnerable SQLite LangChain tool, allowing the LLM to execute raw SQL (RAG Poisoning/Prompt Injection).  
3. **Multi-Agent Handoff (Kiro Bugfix Flow):** Instead of manually hacking the Python files, attendees copy the reproducible attack path provided by AWS Security Agent and feed it into Kiro.**Prompt (in Kiro Spec):** "Create a Bugfix Spec based on this finding from AWS Security Agent: \[Paste AWS Security Agent Finding detailing the SQL injection vulnerability\]."  
4. **Refining the Bugfix Spec (EARS notation):** \* **Current Behavior:** "WHEN an attacker uses prompt injection, THEN the LangChain SQLite tool executes raw DROP TABLE commands."  
   * **Expected Behavior:** "WHEN a query is received, THEN the system SHALL sanitize the input and strictly restrict SQLite tool permissions to read-only views."  
5. **Agentic Execution:** Click **Start Task** in Kiro's tasks.md. Kiro's builder agent autonomously rewrites the LangChain tool definitions and generates property-based tests to prove the fix works.

## **13:30 \- 15:00 | Part 3: Architectural Pivot to AWS & Kiro Agent Hooks**

**Objective:** Shift the DVAIA application from local Ollama/Qdrant to AWS Bedrock and set up Kiro Agent Hooks for proactive security.

### **Detailed Steps:**

1. **Update the Spec for AWS Bedrock:** Attendees open design.md and instruct Kiro to perform an architectural pivot.**Prompt (in Kiro Spec):** "Update the design. We are migrating off local Ollama. Refactor the application to use Amazon Bedrock (Anthropic Claude 3.5 Sonnet) via boto3. Update the tasks.md for this migration."  
2. **Setup Kiro Agent Hooks:** Configure a Kiro Agent Hook to ensure no AWS credentials leak during the refactor.  
   * **Trigger:** fileSave (Pattern: \*\*/\*.py)  
   * **Action (Prompt):** "Review the saved file. If you detect hardcoded AWS credentials, immediately generate a warning and refactor the code to use AWS Secrets Manager."  
3. **Execute the Migration & Deploy:** Attendees click **Start Task**. After Kiro writes the Bedrock integration and AWS CDK/SAM templates, attendees deploy the infrastructure to AWS.

## **15:15 \- 16:45 | Part 4: Incident Response with AWS DevOps Agent**

**Objective:** Experience an operational failure in the newly deployed cloud app, and use **AWS DevOps Agent** to autonomously investigate, map the topology, and suggest a mitigation plan for Kiro to implement.

### **Detailed Steps:**

1. **Simulate a Production Incident:** The instructor triggers a load test script against the attendees' deployed AWS API Gateways. The sudden influx of traffic causes the Lambda functions interacting with Bedrock to hit concurrency limits and timeout, firing a CloudWatch Alarm.  
2. **Autonomous Investigation:** Attendees open the **AWS DevOps Agent Web App** (via the AWS Console).  
   * Notice that the DevOps Agent has automatically detected the CloudWatch Alarm.  
   * View the **Topology Graph**: See how the agent has mapped the API Gateway \-\> Lambda \-\> Bedrock relationship without manual configuration.  
3. **Root Cause Analysis (RCA):** Review the agent's RCA report. The DevOps agent correlates the metrics and logs, identifying that the issue is not a code error, but a throttling/timeout issue due to synchronous Bedrock calls lacking a retry mechanism and insufficient Lambda timeout limits.  
4. **The Mitigation Loop:** AWS DevOps Agent provides a remediation plan (e.g., "Implement exponential backoff using botocore config and increase Lambda timeout to 30s").  
5. **Closing the Loop with Kiro:** Attendees copy the DevOps Agent's mitigation plan and paste it back into Kiro.**Prompt (in Kiro Spec):** "AWS DevOps Agent identified a throttling issue. Update design.md to require exponential backoff for all Bedrock calls and update the IaC template to set Lambda timeouts to 30 seconds. Generate the tasks to apply this."  
6. **Wrap Up:** Attendees click **Start Task** in Kiro to apply the fix, push the code, and resolve the incident.
